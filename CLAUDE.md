## Build, test, format

```
go build ./...
go test ./...
go vet ./...
gofmt -l -w .
```

Every milestone must keep `go build ./...` and `go test ./...` green. Write tests as you go (table-driven). No milestone is "done" with failing or skipped tests.

## Suggested dependencies (keep it minimal)

- SQLite: `modernc.org/sqlite` (pure Go, no cgo — keeps the single-binary build).
- YAML: `gopkg.in/yaml.v3`.
- JSON Schema: `github.com/santhosh-tekuri/jsonschema/v6` (validate against `workflow.schema.json`).
- herdr + `gh`: shell out via `os/exec`; talk to the herdr unix socket via `net` when you wire `events.subscribe` (M4).
- GitHub (later phase only): `github.com/google/go-github`.

Do not add dependencies Phase 1 doesn't need.

## Conventions

- Small interfaces at package boundaries (`exec.ExecutionBackend`, `github.Client`). The engine depends on interfaces, never on herdr/`gh` concretely.
- `context.Context` is the first arg of anything that does I/O or can block; honor cancellation and deadlines (timeouts are config-driven).
- Wrap errors with context (`fmt.Errorf("...: %w", err)`). No panics in the daemon path.
- No global mutable state. The task store is the single writer of task state (one goroutine).
- Parse all herdr/GitHub ids from command output; never hardcode id formats.

## Scope discipline

- Build Phase 1 only. `review`, `approved`, `merging`, `intake`/triage, the scheduler, the MCP server, and context memory are **out** — stub or omit. The engine must parse and validate the full `default-pipeline.yaml`, but only execute the slice path.
- If you find yourself implementing the merge gate or a reviewer, stop — that's a later phase.
